# Polymarket Tail Risk Web 应用设计与实施方案
## 基于 `polymarket_tail_risk_system_v4_research_prd.md`

---

# 1. 目标与范围

本方案将 V4 PRD 转换为可开发的 Web 应用蓝图，目标是：

1. 支撑研究闭环：数据 -> 标签 -> 校准 -> NetEV -> 风控 -> 回测 -> 报告。
2. 支撑运营闭环：候选审核 -> 执行准入 -> 风险状态 -> 审计追踪。
3. 支撑上线闭环：影子运行 -> 分阶段放量 -> 失败可回滚。

本阶段不涉及具体代码实现，仅给出整体结构与实施步骤。

---

# 2. 产品形态定义

该 Web 应用不是单一“交易下单页面”，而是一个“研究与执行前置控制台”，包含三类能力：

1. 研究平台：策略研究员查看样本、校准结果、回测结果、参数版本。
2. 风控平台：风控人员查看组合风险、风险簇、状态机、kill-switch。
3. 运营平台：运营人员处理分类审核、拒绝原因、报告导出、审计追踪。

---

# 3. 总体架构（推荐）

## 3.1 分层架构

建议采用“前后端分离 + 异步任务 + 分析型存储”的分层：

1. 前端展示层（Web Console）
2. 后端 API 层（业务编排）
3. 计算与任务层（数据采集/校准/回测/报告）
4. 数据存储层（OLTP + OLAP + 对象存储）
5. 可观测与审计层（日志、指标、告警、审计）

## 3.2 逻辑模块映射（对齐 PRD 第24节）

1. 数据采集与标准化模块
2. 标签与分类模块
3. 概率校准模块
4. 成本后 NetEV 准入模块
5. 组合风险与仓位模块
6. 状态机与风控状态模块
7. 回测与验证模块
8. 报告与审计模块

## 3.3 架构风格建议

推荐“模块化单体 + 任务工作器”起步，而不是一开始做微服务拆分。

原因：

- V4 仍处于研究主导阶段，规则和模型变化会很频繁。
- 过早微服务会增加部署、调试、数据一致性成本。
- 后续当模块负载明显分化时，再拆分为独立服务。

# 4. 关键用户角色与权限

## 4.1 角色

1. `Researcher`：研究、回测、参数试验、读写研究配置。
2. `Reviewer`：分类审核、灰名单处理、争议标注。
3. `RiskManager`：风控阈值、状态切换审批、冻结恢复审批。
4. `Operator`：任务调度、数据修复、报告发布。
5. `Admin`：权限、系统配置、版本发布。

## 4.2 权限原则

1. 执行准入与风险放开必须分权，不得同一角色独占。
2. Frozen 恢复必须双人审批（至少 `RiskManager + Admin`）。
3. 参数发布必须带版本与审批记录。

# 5. Web 应用信息架构（页面结构）

## 5.1 一级导航

1. 总览 Dashboard
2. 市场宇宙 Market Universe
3. 标签与审核 Tagging & Review
4. 校准中心 Calibration
5. NetEV 准入中心
6. 组合风险 Portfolio Risk
7. 状态机与告警 State & Alerts
8. 回测实验室 Backtest Lab
9. 报告与审计 Reports & Audit
10. 参数与版本 Config & Versions

## 5.2 各页面核心功能

### Dashboard

- 今日候选数、拒绝数、通过率
- 当前风险状态（Normal/Caution/RiskOff/Frozen）
- 最新告警与失败模块状态
- 最近回测结论与上线建议

### Market Universe

- 市场列表与搜索过滤（类别、流动性、到期窗口、争议状态）
- 市场详情（原始规则、价格流、数据质量状态）
- 候选池入池/排除原因追踪

### Tagging & Review

- 自动分类结果与置信度
- 人工审核队列
- 白灰黑名单管理
- 审核历史与对比

### Calibration

- 分层分桶校准结果
- 样本量与区间宽度可视化
- 短窗口 vs 长窗口偏差趋势
- 可交易单元白名单

### NetEV 准入中心

- 候选逐条准入判定
- 成本构成拆解（手续费、滑点、折价）
- 安全边际判定
- 统一拒绝原因码展示

### Portfolio Risk

- 风险簇暴露图
- 单笔/类别/主题/组合限额占用
- 场景压力模拟结果
- 仓位建议与拒绝日志

### State & Alerts

- 市场状态流与信号状态流
- 全局风险状态时间线
- 告警事件与处理动作
- kill-switch 触发与恢复审批

### Backtest Lab

- 策略版本与参数集管理
- 回测任务提交与进度
- 基准对照结果
- 稳健性测试矩阵

### Reports & Audit

- 日报、周报、阶段评审
- 审计导出（候选日志、交易日志、风控日志）
- 不可审计标记与缺失项

### Config & Versions

- 参数分层管理（全局/类别/簇/市场）
- 规则版本与发布时间线
- 变更理由、证据、回滚计划

# 8. API 设计边界（不含代码）

## 8.1 API 分组建议

1. `/markets`：市场查询与详情
2. `/tagging`：分类结果与审核任务
3. `/calibration`：校准单元与区间结果
4. `/signals`：候选信号与准入结果
5. `/risk`：暴露、状态、告警、迁移
6. `/backtests`：回测任务与结果
7. `/reports`：报告查询与导出
8. `/config`：参数与规则版本

## 8.2 API 原则

1. 所有判定返回“结果 + 理由 + 版本号”。
2. 拒绝判定必须返回主原因码和可选次原因码。
3. 状态转换必须包含 `from_state`、`to_state`、触发指标与时间戳。
4. 所有写操作必须写审计日志。

# 9. 技术选型建议（实现导向）

不限定具体框架，但建议满足以下能力：

1. 前端：支持复杂表格、图表、状态流可视化、权限控制。
2. 后端：支持同步 API + 异步任务编排。
3. 存储：同时支持事务数据与分析查询。
4. 消息机制：支持任务队列、重试、幂等。
5. 可观测：日志、指标、追踪、告警一体化。

可采用的典型组合（示例）：

- 前端：React + TypeScript + 图表库
- 后端：Python/TypeScript API 服务
- 任务：异步任务队列与调度器
- 数据库：PostgreSQL（事务）+ 分析库（可选）
- 缓存：Redis
- 对象存储：用于报告与回测产物

# 10. 非功能性需求

## 10.1 可审计性

- 所有关键动作可回放、可追溯、可归因。

## 10.2 一致性

- 判定结果与页面展示必须绑定同一规则版本。

## 10.3 可恢复性

- 任务失败可重试，状态机异常可降级至 Frozen。

## 10.4 安全性

- 分权控制、敏感操作审批、审计不可篡改。

## 10.5 性能

- 列表查询、指标看板、报告导出满足可用时延要求。

# 11. 实施步骤（分阶段）

## 阶段 A：架构落地与数据基建（M1）

目标：

- 建立项目骨架、数据库骨架、任务框架、统一日志规范。

交付：

1. 数据表基础结构
2. 数据采集任务与质量报告
3. Market Universe 页面基础版

验收：

- 能稳定拉取并存储市场与价格快照。
- 能识别并展示关键缺失字段与异常。

## 阶段 B：标签审核闭环（M2）

目标：

- 建立自动分类 + 人工审核闭环。

交付：

1. 标签规则引擎
2. 审核任务流
3. 白灰黑名单管理页面

验收：

- 每个候选市场有可追溯分类结果和审核记录。

## 阶段 C：校准与 NetEV 准入（M3）

目标：

- 打通“校准结果 -> 成本后准入 -> 原因码”链路。

交付：

1. 校准结果计算与展示
2. NetEV 评估模块
3. 拒绝原因码字典与展示

验收：

- 对任意候选可输出“通过/拒绝 + 原因码 + 规则版本”。

## 阶段 D：组合风控与状态机（M4）

目标：

- 将风控从静态规则升级为状态驱动。

交付：

1. 风险簇暴露计算
2. 风险状态管理（Normal/Caution/RiskOff/Frozen）
3. kill-switch 与恢复审批流程

验收：

- 风险阈值触发可自动迁移状态并产生审计记录。

## 阶段 E：回测实验室与报告（M5）

目标：

- 建立可重复的研究验证体系。

交付：

1. 回测任务管理
2. 基准对照与稳健性测试
3. 日报/周报/阶段评审模板落地

验收：

- 可生成结构化回测报告并关联策略版本与参数版本。

## 阶段 F：影子运行与上线评审（M6）

目标：

- 实盘前验证系统稳定性和流程完整性。

交付：

1. 影子运行环境
2. 上线门槛检查面板
3. NoGo/Go 评审流程

验收：

- 全部 DoD 达成后才允许进入小规模真实试运行。

# 12. 并行开发建议（团队拆分）

建议拆为 4 条主线并行，减少相互阻塞：

1. 数据与任务线：采集、清洗、校准、回测任务。
2. 业务与风控线：准入、原因码、状态机、风险逻辑。
3. Web 前端线：控制台页面、图表、审批流。
4. 平台与运维线：部署、监控、权限、审计。

每条线都以“可演示小闭环”迭代，不做大爆炸式交付。

---

# 13. 主要风险与应对

1. 数据源不稳定  
应对：多源冗余、失败重试、数据新鲜度告警。

2. 规则迭代过快导致版本混乱  
应对：强制版本化、变更审批、回滚机制。

3. 回测与在线判定不一致  
应对：统一规则引擎、统一参数源、统一版本号。

4. 风控状态误触发或漏触发  
应对：阈值分级、人工审批兜底、回放复核工具。

5. 页面可视化复杂导致开发拖延  
应对：先交付核心表格与指标，再逐步增强图形交互。

# 16. 决策确认（已采用）

本节作为当前版本的正式决策记录。

## 16.1 已确认技术栈

采用以下统一技术栈：

1. 前端：`Next.js + React + TypeScript + Tailwind CSS + ECharts`  
2. 后端 API：`Python FastAPI`  
3. 异步任务：`Celery + Redis + Celery Beat`  
4. 数据库：`PostgreSQL`（事务主库）  
5. 分析计算：`DuckDB`（研究与回测分析）  
6. 对象存储：`S3 兼容存储（MinIO/S3）`（报告与回测产物）  
7. 鉴权与权限：`JWT + RBAC`  
8. 可观测：`OpenTelemetry + Prometheus + Grafana + Loki`  
9. 部署方式：`Docker Compose`（开发/测试）+ 预留 `Kubernetes` 迁移路径

确认原则：

- 先保证研究与风控闭环完整，再做性能与服务拆分优化。
- 先统一数据与规则版本，再做大规模并发。

## 16.2 已确认团队拆分

采用四条并行主线：

1. 数据与任务线：采集、清洗、任务调度、校准与回测任务编排。  
2. 业务与风控线：准入判定、原因码、状态机、风险约束。  
3. Web 前端线：控制台页面、可视化、审核与配置交互。  
4. 平台与运维线：环境、监控、日志、权限、发布流程。

协作机制：

- 每条主线按“可演示小闭环”迭代交付。
- 每周做一次跨线集成检查，避免接口漂移。
