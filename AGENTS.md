# 项目协作规范

本文件用于约束本仓库内的协作方式、文档方式、编码方式与运行调试方式。
如无特殊说明，后续开发均默认遵守本文件。

## 1. 沟通规范

- 全程使用简体中文进行思考、沟通、说明和文档编写。
- 回答尽量结论先行；如存在假设、风险、取舍，需要明确说明。
- 发现需求、流程、数据结构或架构存在明显问题时，应及时提醒，不隐瞒风险。

## 2. 编码与乱码规范

- Markdown、TypeScript、JavaScript、JSON、YAML、SQL、Shell 脚本等文本文件，统一使用 UTF-8 编码。
- 优先使用 UTF-8 无 BOM，避免乱码、兼容性问题和补丁工具误判。
- 前端界面、文档、日志、导出说明中不得出现乱码；如发现乱码，优先先修复编码问题，再继续功能开发。
- 引入中文文案时，要同时考虑字体链、浏览器显示和终端输出差异。

## 3. 文档规范

- `.md` 文档默认使用中文编写。
- 当前仓库正式文档统一写入 `doc/` 目录。
- 用于讨论、评审、方案比较的文档，优先写入 `discuss/`；若项目尚未建立该目录，则暂放 `doc/`，并使用清晰文件名标识用途。
- 涉及项目结构、启动方式、接口约定、数据库初始化、主流程变化时，需要同步更新 `README.md`。
- 每天开发结束前，应更新 `doc/dev-progress.md`，记录当天完成项、验证结果和下一步计划。

## 4. 架构与代码规范

### 4.1 基本原则

- 优先保证主链路可跑通，再做局部优化；禁止脱离业务闭环的过度设计。
- 优先保持单一职责、清晰分层、低耦合。
- 发现坏味道时，应主动指出，并在改动范围较大时先与你确认优化方式。

### 4.2 规模控制目标

以下是“强约束目标”，用于控制复杂度；如因阶段性原因暂时超出，应优先考虑拆分，而不是长期放任。

- 对 Python、JavaScript、TypeScript 等动态语言文件：单文件尽量不超过 300 行。
- 对 Java、Go、Rust 等静态语言文件：单文件尽量不超过 400 行。
- 单层目录中的文件数尽量不超过 8 个；超出时优先拆分为更清晰的子目录。

### 4.3 需要持续警惕的坏味道

开发、阅读、评审代码时，都需要主动识别并避免以下问题：

- 僵化：一点小改动引发连锁修改。
- 冗余：相同逻辑多处重复，实现不一致。
- 循环依赖：模块之间互相缠绕，难以测试和复用。
- 脆弱：改一处坏多处。
- 晦涩：命名、结构、意图不清，阅读困难。
- 数据泥团：多个参数总是成组出现，却没有抽象成对象。
- 不必要的复杂性：用过重方案解决简单问题。

## 5. 强类型与数据结构规范

- 在 TypeScript、Python 等支持类型的场景中，优先定义清晰的数据结构与类型。
- 尽量避免 `any`、未约束的裸 `json`、未结构化字典。
- 若确实需要使用 `any`、未结构化对象或动态数据，应满足以下要求：
  - 先判断是否可以通过类型、接口、DTO、类型守卫解决。
  - 若仍无法避免，应把使用范围收窄到最小。
  - 对高风险用法，先与你确认。
- 公共数据结构优先沉淀到共享层，避免前后端重复定义、语义漂移。

## 6. 运行与调试规范

- 运行、调试、联调、构建、测试等操作，统一优先通过 `scripts/` 目录下的脚本执行。
- 当前开发环境以 Windows 为主，因此脚本入口优先支持 `.ps1`；如有跨平台需要，可补充 `.sh`。
- 禁止长期依赖零散手敲命令；一旦某类操作会被重复使用，就应沉淀为 `scripts/` 下的脚本。
- 若脚本执行失败，应先修复脚本或相关流程，再继续使用统一脚本。
- 运行与调试前，应尽量配置文件日志输出，并统一落到 `logs/` 目录；若当前阶段尚未完成，应尽快补齐。
- 在仓库尚未补齐 `scripts/` 之前，允许临时使用现有 `package.json` 脚本进行过渡，但后续需要补齐统一脚本入口。

## 7. Python 规范

- Python 虚拟环境目录统一使用 `.venv`。
- Python 相关依赖管理、运行与调试优先使用 `uv`。
- Python 项目根目录保持简洁，只保留必要文件。
- `main.py` 或等价入口文件保持简洁，避免堆积业务逻辑。

## 8. React / Next.js / TypeScript / JavaScript 规范

- 前端默认优先使用 TypeScript，而不是纯 JavaScript。
- React 项目优先使用 React 19。
- 如引入 Next.js，优先使用 15.4 或更高版本。
- 如引入 Tailwind CSS，优先使用 v4。
- 新建前端包、新建工具包默认优先采用 ESM。
- 现有项目若已使用 CommonJS，不强制立即重构；但新增模块和新建子项目应避免继续扩大 CommonJS 使用范围。
- 涉及表单、接口、状态流转时，优先定义明确类型、状态枚举和转换规则。

## 9. 现阶段对本项目的补充说明

- 当前正式文档目录为 `docs/`，这是本仓库的有效标准。
- 当前仓库尚未建立完善的 `scripts/` 体系，后续开发中需要逐步补齐。
- 当前后端仍存在 CommonJS 历史配置，短期允许保持可运行；后续若做模块制式升级，需要单独规划，不在日常小改中顺手硬切。
- 当前第一优先级仍然是：保证任务计划、数据接入、审批、报表这条业务主链路真实可跑通。

## 10. 执行要求

- 后续每次开发、评审、修复、重构，默认遵守本文件。
- 如果本文件与仓库现实情况冲突，应优先说明冲突点，再给出兼容处理方案，不允许默默忽略。
- 如果本文件需要继续细化，应直接修改本文件，而不是把规则分散到多个角落。
