# Polymarket Tail Risk NO 策略
## 可执行的详细开发计划

## 1. 文档目标

本文档不是策略口号，而是一份面向开发执行的 MVP 计划。目标是把“系统性买入被高估的小概率事件的 NO”这件事，落成一个可运行、可回测、可风控、可审计的自动化交易系统。

本计划重点解决四个问题：

1. 做什么，不做什么。
2. 系统要拆成哪些模块。
3. 按什么顺序开发，怎么验收。
4. 上线前必须满足哪些风控和测试条件。

---

## 2. 策略定义

### 2.1 核心思路

策略只做一件事：

> 在 Polymarket 上，筛选临近结算、NO 价格极高、且语义上更偏“数值型/时间型判断”而非“现实世界灾难型事件”的市场，买入 NO。

### 2.2 策略假设

该策略建立在以下假设上：

1. 市场中存在被高估的小概率事件。
2. 越接近结算日，部分离谱定价更容易收敛。
3. “数值型/时间型市场”相比“黑天鹅/灾难型市场”，更适合做高赔率 NO。
4. 黑天鹅无法避免，只能通过仓位控制和分散化降低致命损失。

### 2.3 策略本质

该策略本质上接近“卖极端风险保险”，特点如下：

1. 短期可能表现稳定。
2. 胜率理论上较高，但单次极端亏损不可忽略。
3. 成功关键不是追求零回撤，而是保证单次或连续极端事件不会击穿资金曲线。

---

## 3. 范围界定

### 3.1 MVP 要做的事情

MVP 只覆盖以下能力：

1. 拉取市场列表与基础行情。
2. 计算候选市场的 NO 价格、剩余天数、文本标签。
3. 根据固定规则筛选候选市场。
4. 做基础风控校验。
5. 支持模拟下单和真实下单两种模式。
6. 用 SQLite 持久化市场快照、信号、订单、持仓、风控事件。
7. 输出基础报表和每日运行日志。

### 3.2 MVP 不做的事情

MVP 明确不做以下内容，避免项目失控：

1. 不做多策略。
2. 不做复杂机器学习模型。
3. 不做自动调参平台。
4. 不做高频交易。
5. 不做多交易所适配。
6. 不做前端管理后台，先以 CLI 和日志为主。
7. 不做自动新闻抓取和事件真伪判断。

## 4. 交易规则

### 4.1 入场规则

候选市场必须同时满足以下条件：

1. `no_price >= 0.97`
2. `days_to_resolution <= 7`
3. 问题文本命中至少一个白名单模式
4. 问题文本不能命中黑名单模式
5. 市场状态可交易，且流动性高于最低要求
6. 当前没有重复持仓，或未超过该市场允许加仓上限

### 4.2 白名单方向

优先考虑以下类型：

1. 数值阈值类
2. 时间截止类
3. 统计结果类
4. 明确日期结算类

初始白名单关键词可从以下模板开始：

```python
WHITELIST = [
    "price >",
    "price <",
    "by date",
    "before friday",
    "this week",
    "odds",
]
```

### 4.3 黑名单方向

避免以下高尾部风险主题：

1. 战争
2. 袭击
3. 军事冲突
4. 暗杀
5. 死亡
6. 地震
7. 空难或重大事故
8. 市场崩盘
9. 紧急事件

初始黑名单关键词：

```python
BLACKLIST = [
    "war",
    "attack",
    "military",
    "assassination",
    "death",
    "earthquake",
    "crash",
    "collapse",
    "emergency",
]
```

### 4.4 暂不做的复杂规则

以下规则先不进入 MVP，但应保留接口：

1. 文本语义分类模型
2. 市场历史波动过滤
3. 做市深度评分
4. 相关新闻一致性验证
5. 同主题相关市场联动风险识别

## 5. 风控原则

### 5.1 总体原则

风控优先级高于收益。任何无法解释、无法追踪、无法限制损失的交易都不执行。

### 5.2 MVP 风控参数

建议初始值如下，后续通过回测和模拟盘调整：

```python
MIN_NO_PRICE = 0.97
MAX_DAYS_TO_RESOLUTION = 7

MAX_CAPITAL = 1000
MAX_CAPITAL_USAGE = 800

MAX_POSITIONS = 30
MAX_POSITION_PER_MARKET = 20
MAX_NEW_TRADES_PER_DAY = 5

DEFAULT_ORDER_SIZE = 10
MIN_MARKET_LIQUIDITY = 500
```

### 5.3 风控检查项

每次下单前必须按顺序校验：

1. 总可用资金是否足够。
2. 当日新增交易数是否超限。
3. 当前总持仓数是否超限。
4. 该市场持仓金额是否超单市场上限。
5. 是否已存在未完成订单。
6. 市场是否已停止交易或接近停止交易。
7. 市场流动性是否低于阈值。
8. 市场是否命中人工禁入名单。

### 5.4 退出与清理

MVP 先做最简单退出逻辑：

1. 市场结算后自动归档持仓。
2. 对成交但未结算的仓位持续跟踪。
3. 不做主动止盈止损，先以到期结算为主。

后续版本可新增：

1. 达到目标收益率提前平仓。
2. 流动性恶化时主动减仓。
3. 临近结算时的异常盘口退出。

### 6.2 模块职责

#### `data`

负责从 Polymarket 或其对应接口拉取市场数据，并统一格式。

输出统一字段至少包括：

```python
{
    "market_id": str,
    "question": str,
    "yes_price": float,
    "no_price": float,
    "liquidity": float,
    "volume": float,
    "close_time": datetime,
    "status": str,
    "days_to_resolution": int,
}
```

#### `strategy`

负责对市场做规则过滤、打标签、生成交易信号。

#### `risk`

负责对交易信号做资金和仓位限制，输出是否允许下单及拒单原因。

#### `execution`

负责实际下单，支持两种模式：

1. `paper` 模拟盘
2. `live` 实盘

#### `storage`

负责持久化，保证每次扫描、每次信号、每次订单都有记录。

#### `reports`

负责指标统计、日报输出、回测结果汇总。

## 8. 核心流程

### 8.1 单次扫描流程

```text
拉取市场数据
-> 数据清洗与标准化
-> 价格过滤
-> 时间过滤
-> 文本白黑名单过滤
-> 风控校验
-> 生成信号
-> 执行下单或模拟下单
-> 写入数据库
-> 输出日志和报告
```

### 8.2 核心伪代码

```python
def run_once():
    markets = market_feed.fetch_open_markets()

    normalized = [normalize_market(m) for m in markets]

    for market in normalized:
        snapshot_repo.save(market)

        signal = strategy.evaluate(market)
        signal_repo.save(signal)

        if not signal.should_trade:
            continue

        risk_result = risk_manager.check(signal, market)
        risk_repo.save(risk_result)

        if not risk_result.allowed:
            continue

        order_result = trader.buy_no(
            market_id=market["market_id"],
            size=settings.DEFAULT_ORDER_SIZE,
            mode=settings.RUN_MODE,
        )
        order_repo.save(order_result)
```

## 9. 开发阶段规划

以下阶段按“先可跑，再可验，再可实盘”的原则推进。

### Phase 0：需求冻结与技术验证

目标：

1. 确认可用数据源和交易接口。
2. 确认 Polymarket 接口认证方式、频率限制、字段结构。
3. 明确模拟盘模式的行为定义。

交付物：

1. 接口调研结论。
2. `.env.example` 所需环境变量列表。
3. 一份字段映射文档。

验收标准：

1. 能成功拉取一批开放市场数据。
2. 能拿到市场 close time、价格、流动性等关键字段。
3. 能明确真实下单接口是否可直接接入。

风险点：

1. 官方接口权限限制。
2. 字段不稳定或多接口拼装。
3. 实盘下单需要链上签名或额外中间层。

### Phase 1：数据层与基础脚手架

目标：

1. 初始化项目结构。
2. 完成配置、日志、数据拉取、标准化模型。
3. 建立 SQLite 数据库。

开发任务：

1. 搭建项目目录与依赖。
2. 实现配置加载和环境变量校验。
3. 实现 `market_feed.py`。
4. 实现市场标准化模型。
5. 初始化 `schema.sql` 和数据库连接层。
6. 实现市场快照落库。

交付物：

1. 可执行的 `scripts/run_once.py`
2. `market_snapshots` 正常写入
3. 基础日志输出

验收标准：

1. 单次运行可拉取并存储 100+ 市场。
2. 无关键字段缺失导致的崩溃。
3. 数据库可重复初始化。

### Phase 2：策略过滤器

目标：

1. 实现价格、时间、语义三层过滤。
2. 输出标准化信号对象。

开发任务：

1. 实现 `is_price_qualified`
2. 实现 `is_time_qualified`
3. 实现 `is_whitelisted`
4. 实现 `is_blacklisted`
5. 实现 `tail_no.py` 的 `evaluate_market`
6. 将过滤结果存入 `signals`

交付物：

1. 策略评估器
2. 白名单/黑名单规则配置
3. 信号落库能力

验收标准：

1. 每个市场都能得到明确的通过/拒绝原因。
2. 白黑名单规则可通过配置更新，无需改代码主逻辑。
3. 单元测试覆盖主要路径。

### Phase 3：风控管理器

目标：

1. 实现最小可用风控。
2. 保证不会在无约束条件下连续下单。

开发任务：

1. 实现资金占用检查。
2. 实现最大持仓数检查。
3. 实现单市场仓位上限检查。
4. 实现单日新增交易数检查。
5. 实现手工禁入名单。
6. 风控结果落库。

交付物：

1. `risk/manager.py`
2. `risk_events` 持久化
3. 拒单原因可追踪

验收标准：

1. 任一风控条件触发时不会下单。
2. 拒单原因清晰且可审计。
3. 同一市场重复扫描不会无限重复开仓。

### Phase 4：执行层

目标：

1. 实现模拟盘交易。
2. 预留实盘接口适配层。

开发任务：

1. 实现 `simulator.py`
2. 定义 `broker.py` 抽象接口
3. 实现 `trader.py`
4. 记录订单和持仓状态
5. 实现结算后自动归档

交付物：

1. `paper` 模式完整可跑
2. 订单与持仓表联动
3. 基础结算处理

验收标准：

1. 模拟模式下能生成完整订单链路。
2. 持仓状态能随市场结算变化更新。
3. 任意异常不导致数据库状态错乱。

### Phase 5：回测与模拟运行

目标：

1. 用历史市场数据验证策略方向是否成立。
2. 确定初始参数区间。

开发任务：

1. 设计回测输入格式。
2. 回放历史市场快照。
3. 计算胜率、平均收益、最大回撤、单次最大亏损。
4. 比较不同阈值，如 `0.95/0.96/0.97/0.98`。

关键关注指标：

1. 胜率
2. 平均盈亏比
3. 最大回撤
4. 最大连续亏损次数
5. 每日新增交易数分布
6. 资金利用率

交付物：

1. 回测脚本
2. 参数对比报告
3. 风险结论摘要

验收标准：

1. 能输出可复现的回测结果。
2. 能比较至少 4 组核心参数。
3. 能识别最危险的市场类别和关键词。

### Phase 6：小资金模拟实盘

目标：

1. 用 `paper` 模式连续运行 7 到 14 天。
2. 验证稳定性、日志、数据完整性。

运行要求：

1. 每日定时扫描。
2. 每次运行输出候选市场和拒绝原因。
3. 每日汇总仓位、信号数、风控拦截数。

验收标准：

1. 连续运行期间无严重崩溃。
2. 日志和数据库记录完整。
3. 不出现重复下单、脏数据、持仓错账。

### Phase 7：小资金真实交易

前提条件：

1. 回测结果可接受。
2. 模拟盘稳定运行至少 1 周。
3. 实盘下单接口验证完成。
4. 具备人工紧急停机手段。

建议初始资金：

1. `100-300 USDT`

上线限制：

1. 初期只开 `live + low frequency`
2. 单笔固定小仓位
3. 每日新增交易数压到最低
4. 首周必须人工复核交易日志

验收标准：

1. 能完成真实下单、订单记录、持仓归档。
2. 异常时可手动停机并恢复。
3. 资金与账面记录能核对一致。

## 12. 关键决策与建议

### 12.1 为什么阈值建议从 `0.97` 起步

原始方案使用 `NO > 0.95`，但从风险角度看，MVP 初期更适合使用更保守阈值：

1. 可减少伪低概率事件进入样本。
2. 可降低初期误判造成的连续亏损。
3. 有助于先验证系统稳定性，而不是盲目追求信号数量。

### 12.2 为什么先不做主动止盈止损

因为当前最核心问题还不是优化收益，而是先验证：

1. 策略是否真的能稳定筛出合理标的。
2. 数据和执行链路是否可靠。
3. 结算结果是否可完整回收。

### 12.3 为什么必须记录全部拒单原因

因为这个策略未来能否优化，主要依赖以下分析：

1. 哪类市场被过滤掉最多。
2. 哪类市场经常卡在风控。
3. 哪些白名单带来好收益，哪些带来坏收益。

没有这些记录，后续优化将几乎无法进行。

## 15. 最终结论

这个项目最重要的，不是“尽快自动下单”，而是先把下面三件事做扎实：

1. 过滤规则是否足够保守。
2. 风控是否能有效限制尾部损失。
3. 数据、订单、持仓是否全链路可追踪。

只有这三件事成立，这个系统才有资格进入小资金实盘阶段。

一句话概括：

> 这不是套利机器人，而是一台“系统化筛选并承受尾部风险”的交易系统。开发时必须把风控、审计和可复盘能力放在收益之前。
